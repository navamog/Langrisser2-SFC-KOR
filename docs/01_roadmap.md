# 한글패치 로드맵 & 방법론

## 확정된 사실 (정적 분석 완료)

| 항목 | 결과 |
|------|------|
| 매퍼 | LoROM (FastLoROM, mapmode 0x30) |
| 롬 크기 | JP 2MB → 확장 시 3MB (4MB까지 여유) |
| 텍스트 인코딩 | **표준 Shift-JIS 아님 → 커스텀 테이블** (한자는 자체 폰트 인덱스) |
| 폰트 위치 | 0x110000~ 은 초상화/배경 그래픽. **폰트는 압축된 것으로 추정** (평문 미발견) |
| 영문 텍스트(LangFanDood) | 확장영역 0x200000+, `F7 xx` **사전 압축** |
| 코드 영역 | 0x000000~0x04FFFF 부근 (0xA9=LDA 등 65816 밀집) |

## 정적 분석의 한계 (정직한 진단)

이 게임은 **텍스트=커스텀 인코딩 + 폰트=압축** 조합이라, 바이너리만 훑어서
대사 위치·인코딩·포인터를 확정하는 것은 신뢰도가 낮습니다. 이런 타이틀은
**에뮬레이터 디버거로 실행 중 추적**하는 것이 업계 표준입니다.

## 권장 다음 단계: 에뮬레이터 런타임 추적 (Mesen2 / bsnes-plus)

목표: "화면에 뜬 대사 한 줄"이 롬의 어느 바이트에서 왔는지 역추적.

### A. 폰트 렌더 루틴 → 인코딩 테이블 확보
1. Mesen2에서 `base_kr.smc`(JP) 실행, 대사창이 뜨는 장면까지 진행.
2. **PPU/VRAM 쓰기 브레이크포인트**를 폰트 타일이 올라가는 VRAM 주소에 설정.
3. 브레이크가 걸리면 콜스택을 거슬러 올라가 **텍스트 읽기 루프**를 찾음.
4. 그 루프가 읽는 롬 주소 = 대사 데이터. 읽는 바이트 ↔ 화면 글자 대응 →
   `table/kr.tbl` 작성 (예: `82A0=あ` 형식).

### B. 포인터 테이블 확보
1. 대사 데이터 시작 주소를 로드하는 코드(보통 `LDA long,X` / 포인터 인덱싱)를 관찰.
2. 포인터 테이블 위치·형식(2바이트 오프셋 + 뱅크? 3바이트 롱?) 기록 → `docs/pointers.md`.

### C. 폰트 압축 해제
1. 폰트 타일을 VRAM에 올리기 직전의 **디컴프레션 루틴** 확인.
2. 압축 포맷 식별(LZ 계열 다수) → 디컴프/리컴프 도구 작성.
3. 또는 폰트를 RAM에서 덤프해 글리프 배열·크기(16x16 등) 확정.

## 병행 가능 작업 (추적과 무관하게 지금 할 수 있는 것)

- **영문 대사 텍스트 추출**: LangFanDood 롬의 `F7` 사전 압축을 풀어 영문 스크립트
  확보 → 번역 원문/대응표로 사용. (사전 테이블도 롬 내에 있으므로 추적으로 확보)
- **한글 폰트 글리프 준비**: 게임 폰트 크기 확정 후, 필요한 완성형 한글 글리프를
  비트맵으로 렌더(도트 폰트). 우선 사용 빈도 높은 음절부터.
- **재삽입 파이프라인 골격**: `.tbl` 기반 덤프/삽입기, 포인터 재계산기 (인터페이스
  먼저 잡아두면 표 확보 즉시 연결).
